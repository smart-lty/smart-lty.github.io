Git 使用中踩的一个大坑

March 30, 2022 in project

git真的能让人血压拉满呜呜呜

昨晚，发生了一件差点让我睡不着的事情（指血压过高）。

究竟是怎么回事呢？请往下看。

background

这件事情发生的背景是，我打算用hugo搭建个人网站并部署在github pages上。而想将个人网站部署到github pages上，需要以下几个步骤：

  1. 在github中新建{username}.github.io仓库。
  2. 将网站文件传到此仓库中。
  3. 完成部署。 当然其实以上的3个步骤描述的并不清晰，因为这里只是当时的我google出来的方法。那问题出在哪一步呢？

1. 新建仓库

这一步没什么好说的，在github中点击New Repositories即可。

Creating Repositories

2. 上传至github

这一步就是问题的关键所在了。首先，我在google搜索部署网页的方法时，有一条网页提到，只需要将hugo生成的静态文件发送至github即可。于是呢，为了文件不发生冗余，我做了如下事情：

  • hugo -d docs 将个人搭建的网站生成静态文件，放置于docs/目录下。
  • git add docs, git commit -m"initial commit", git push -u origin master一气呵成，将文件上传至github。

问题来了，当我发现这样做部署网页不成功，并且我还做了一些修改时，我该怎么办呢？

聪明的我想到了，可以用git的reset功能呀！

于是我使用了git reset --hard HEAD^, 浑然没考虑过其后果。什么后果呢？

先前提到，在commit之前，我仅add了hugo生成的静态文件，i.e. docs/文件。也就是说，在.git文件中，仅仅只保留了docs/这个文件夹！那么这个时候使用reset，后果只有一个————除docs/以外的文件，全部丢失！！！

我写了半天的网站，一下子全没了，这我能忍？

我立马祭出杀招————git reflog

嘿，您猜怎么着？！因为没有commit，git reflog完全找不到丢失的文件！

这时候我在思考一件事：为什么我不会抽烟。

好了，既然我把这件事写成了blog，那么最后我一定还是解决了的。各位看官看到这，是不是很好奇，文件都没commit，.git文件没有相关记录，那还怎么找呢？

救世主：git fsck --lost-found

这个命令让我找到了我主要修改的两个文件的内容（部分图片资源仍然丢失了）。于是我苦逼地hugo new site {***}重新创建网站，将内容复制了过来。网站终于恢复如初。

Conclusion: 不确定的时候，千万别乱用git reset!!!

3. 部署到github pages

这一步相对来说就简单多了。运行命令hugo来生成public文件夹。最后将整个project都上传到第一步中新建的项目中。注意要将config.toml中的baseURL修改为"https://{username}.github.io/"。部署到github之后，点击Action可以看到网页正在部署，大约1min之后，在浏览器输入"https://{username}.github.io/"就可以访问个人主页啦！

至此，大功告成！
